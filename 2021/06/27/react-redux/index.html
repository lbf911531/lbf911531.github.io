    react-redux初探

    发表于 2021-06-27 更新于 2021-06-29 分类于 前端

    1.定义

    redux本身是一个状态管理库，个人将它理解为在js中是一个更大的作用域，定义在该作用域中有变量与用于操作变量的函数。它能够使得在局部作用域中的各种组件访问到共有成员。

    2.注意

    redux 不局限于react使用，尽管vue有vuex，但似乎redux也可以用于vue。

    3.使用

    3-1. store

    // App.js:
import { Provider } from 'react-redux';
import { createStore } from 'redux'

const store = createStore(reducer) // reducer这个东西会在下面说明

function App() {
  return (
    <Provider store={store}>
      <App />
    </Provider>
  )
}

    在react中，首先也是最重要的一点，需要在组件外，包裹一层Provider,并将申明的store注入
    这样一来Provider下的App组件及其所有子组件都能拿到或者说访问store的state

    ps: store在整个项目中有且只有一个，但是它可以是多个子store的集合体，确切的说是多个子reducer，每个字reducer内有一个state，然后通过redux提供的combineReducers方法，将各个reducer合并

    那么 combineReducers 合并reducer又是怎么用的呢?

    很明显这种情况下，在项目中的目录架构必然会形成这个格式

    ├── src
  ├── store
    ├── index.js
    ├── reducer1
      ├── index.js
      └── action.js
    ├── reducer2
      ├── index.js
      └── action.js
  ├── App.js
  └── containers
    ├── demo1.js
    └──demo3
      ├── index.js
      └── style.css

    我们会申明一个store文件夹，在store文件夹下有一个index文件，而它会是根state

     // store/index.js
  import { createStore, combineReducers } from 'redux';
  import * as Reducer1 from './reducer1';
  import * as Reducer2 from './reducer2';
  import * as Reducer3 from './reducer3';
  import * as Reducer4 from './reducer4';

  export const store = createStore(
    combineReducers({...Reducer1,...Reducer2,...Reducer3,...Reducer4})
  )
  // 通过combineReducers 合并并返回一个大的reducer作为createStore的入参

    然后，你就可以在reducer1，reducer2，reducer3…这些文件里愉快的定义你的action了

    到这里 redux的三个基本元素 store就暂时告一段落了。

    3-2. reducer

    作为使用redux中最重要的一环，它的本质是一个纯函数。而所谓的纯函数，是指不依赖外界参数，不会产生副作用的函数。意思诙谐一点就是：1.任你执行，本函数内的操作不会干预与本函数同一作用域下的其他东西无关，本函数不会去打扰人家，即使你想，本函数也不允许；2. 任你输入，只要入参不变，我始终会输出相同结果。

    • 体现到代码里就是：
      1. const state = {propA: {}, propB: 1};
      2. const stateX = function pureFunc(state,propAX) { return {…state, propsA: propAX} };

    但是，有很重要的一点，一个组件我会有很多交互事件，那么我要怎么在一个reducer函数里区分不同的action呢？
    if…else/ switch…case 是一个合适的选择

    // store/reducer1/index.js
import {
  ADD_NUM,
  PUSH_NAME_LIST,
} from './action'; // 这是为了避免日后 字符串随意改值而减少代码改动量，将常量抽出放入一个action的文件内
const initState = {
  num: 0,
  list: [{ name: '张三', id: '1' }, { name: '李四', id: '2' }],
}

export const myFirstReducer = (state = initState, action) => {
  switch (action.type) {
    case ADD_NUM:
      return ({ ...state, num: state.num + action.payload })
    case PUSH_NAME_LIST:
      const tempList = [...state.list];
      tempList.push(action.payload);
      return { ...state, list: [...tempList] }
    default:
      return state;
  }
}

    可以看到上述代码片段，让我们将注意力放到以下几点

    1. state,action两个入参
    2. case 后面跟着的值
    3. default

    一： 由于纯函数只依赖入参，和外界其他变量没有关系。放到具体场景里，我们需要取state里的值并根据不同的动作
    得到不同的state。这样就必然需要将state传入，然后再抛出。action对象里则拥有着我们对应的动作，和该动作
    需要对state不同的变化。
    二： 在某些情况下我们的case值需要做出一些改变，可能这个时候叫这个值，下一刻就叫另一个。这种情况下我们为了
    避免去找reducer函数直接修改，而选择将它们提取出来。单独声明 一个个常量，case后面紧跟这些常量。然后在需要改变的时候去改变定义的常量就好了。（ps:其实不仅仅在reducer中会用到这些常量）
    三： 不论你改变或者不改变，都必须要返回。

    3-3. dispatch

    在了解dispatch之前我们先画一张图：

                  dispatch(action)    (previousState,action)
actionCreators ==========>  store =========> reducers
                              ||  <=========          
                      state   ||    newState
                              ||
                              || connect(mapStateToProps)
                            Components

    actionCreators： 我们定义了各种行为常量
    store：这是组件直接面对的层面，存放了state
    reducer： 在这里，我们根据不同的行为常量返回不同的state
    Components ： 组件需要访问公共的state，同时发出交互事件命令。

    ———– 这是一种发布/订阅模式 ————-
    (ps: 至于什么是发布/订阅模式，我们以后讨论，这里只需要了解一个概念，就是 订报，送报； 用户（组件）通过报纸社（redux）提供的 订报服务 （Provider，connect）关联，然后等当天日报印刷后（store中的相关属性修改操作）派报童（dispatch）送给用户（组件），用户（组件）阅览日报（新的属性），就可以做出相应的变换（渲染） )

    很明显，联系它们的是 state和能够改变state的dispatch函数
    dispatch函数携带一个action参数，发布一系列的操作，到reducer中，然后所有订阅了这个state的组件就同时会感知到操作带来的变化。

    而action,这个参数最重要的两个属性，type,payload。type对应的行为常量，payload对应的该行为所要的变化凭依。

    // containers/demo3/index.js
import React from 'react';
import { connect } from 'react-redux';

function Components(props) {
  function addNum() {
    const { dispatch } = props;
    dispatch({
      type: ADD_NUM,
      payload: 1,
    })
  }
  return (
    <div>
      <div>{props.state1.num}</div>
      <div><button onClick={addNum}>click me and add</button></div>
    </div>
  )
}
function mapStateToProps (state, ownProps) {
  return {
    state1: state.myFirstReducer, //  state.myFirstReducer这个变量请和reducer的名字保持一致。
  }
}

export default connect(mapStateToProps)(Components);

    补充一下： react-redux 提供的connect方法，使我们的组件被包裹，加工为高阶组件。它提供了四个参数
    第一个使mapStateToProps,能够让我们从store拿到指定的reducer及其所拥有的state。。

    mapStateToProps

    顾名思义， 这个东西是将 state注入到当前组件的props里，方便组件从props里直观获取到redux对应reducer state里面的值。

    套用网上的话来说：mapStateToProps是一个函数。它的作用就是像它的名字那样，建立一个从（外部的）state对象到（UI 组件的）props对象的映射关系。作为函数，mapStateToProps执行后应该返回一个对象，里面的每一个键值对就是一个映射。

    mapStateToProps有两个参数 (state,ownProps)

    第一个不必多说，只要你打印一下，一切就明了。
    第二个的存在感就相对薄弱，一般不怎么用它。使用ownProps作为参数后，如果容器组件的参数发生变化，也会引发 UI 组件重新渲染。这是阮大神的解释，但看上去，是的，没什么感觉。作为自己拥有的props，它肯定来自于父组件传递的属性，如果是必要重渲染，那么我们肯定会在生命周期内引用它达到某些行为。无需添加ownProps。

    mapDispatchProps

    mapDispatchToProps是connect函数的第二个参数，用来建立 UI 组件的参数到store.dispatch方法的映射。也就是说，它定义了哪些用户的操作应该当作 Action，传给 Store。它可以是一个函数，也可以是一个对象。

    用我自己的话来说，这个东西实际上就是将你的action和dispatch这些整合到一起。
    原本你要发布，需要这样写： dispatch({type: action, extraParams:{}});
    那么现在就是：mapDispatchProps 帮你多做了一步 const func = dispatch({type: action, extraParams:{}})。而你只需要 在用到的时候 这样写： func();

    网上说 mapDispatchProps 使用方法有好几种，但在我看来。可以归纳成两种，函数或者对象。

    // reducer2/index
import axios from 'axios';

const initialState = { 
  projects: [],
  taskMarketingList: [],
  page:{},
  loading: false,
  status: '',
};

export default function taskMarketingResult(state = initialState, action) {
  switch (action.type) {
    case "SELECT_FETCH_SUCCESS":
      return {
        ...state,
        promise: action.promise,
      };
    case "ADMIN_FETCH_FAIL":
      return{
        ...state,
        loading: false,
        status: "fail",
      };
    default:
      return state;
  }
}

export function initSelects() {
  return {
    type: "SELECT_FETCH_SUCCESS",
    promise: () => {console.log(1);return axios.get("http://news-at.zhihu.com/api/2/news/latest")}
  };
}

// Components
import { bindActionCreators } from 'redux';
import React, { useEffect } from 'react';
import { connect } from 'react-redux';
import { initSelects } from './reducer2';

function Components(props){
  useEffect(() => {
    props.dispatch1() 
  },[])
} // useEffect不用在意，它是hook语法，useEffect(() => {},[]) 等价于componentDidMount

const mapDispatchToProps = (dispatch, ownProps) => {
  return {
    dispatch1: bindActionCreators(initSelects,dispatch)
  }
}
export default connect(()=>{},mapDispatchToProps)(Components)

或者是下面这种

export default connect(()=>{},{initSelects})(Components)

    dva redux使用

    dva，可以类别于vuex，它帮我们封装了redux，在项目初始化的时候，自动生成了一个models文件夹
    当然也可以在你自己的文件夹内新建一个models。

    确切的讲，编译的时候，dva的机制会自动查询到名为models的所有文件夹，然后统合models下的文件，
    类比到react-redux中，即为combineReducers。所以有一点需要注意，每一个models下的文件，第一
    名字最好不要重复；第二，该文件内的namespace属性值绝对不要重复

    
// models/cache.js
export default {
  namespace: 'cache', // 命名空间，
  state: { // 当前reducer所管理的 小 store
    cache: {},
  },
  reducers: {
    setCache(state,action) {
      return {...state,cache: action.payload },
    }
  },
}

// 组件
import { connect } from 'dva';

function DemoComponent(props) {
  function operator () {
    const { dispatch } = props;
    // 命名空间/函数名 ，类似于找到 cache对象下的 setCache方法，经由dispatch
    // 压入 事件队列， 等待执行 cache.setCache(payload)
    dispatch({
      type: 'cache/setCache', 
      payload: 'demo' + new Date(),
    })
  }
  return (
    <div>
      <button onClick={operator}> click me</button>
      <div>{props.cache}</div>
    </div>
  )
}

/* state: 最大的也是统合后的store所代表的对象 */
function mapStateToProps(state) {
  return {
    cache: state.cache.cache, // key随意取名， state后面紧跟自定义的命名空间名， 再之后即时该空间内state的某一个属性，这里可以省略为 state.cache ，将整个抛出
  }
}
export default connect(mapStateToProps)(DemoComponent)
